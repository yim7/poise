# Mainnet 单实例值守路线设计

本文档定义 `grid-platform` 在 `2026-03-22` 之后的下一阶段路线。目标不是继续扩客户端形态，而是让系统具备单实例 mainnet 自动运行与本地值守的资格。

本设计明确采用“实例感知，单实例验收”的边界：

- 平台层面允许同一环境承载多个实例
- 当前主线只要求一个已配置实例达到 mainnet 可持续运行标准
- 不把多实例运营能力、远程操控和 Web UI 作为当前主线

## 1. 目标

- 让一个已配置实例具备进入 mainnet 自动运行的最低安全资格
- 让该实例在本地值守场景下具备 `7x24` 运行所需的健康判定、告警和恢复能力
- 让当前网格策略更适合长期实盘运行，而不是只在 replay / paper / testnet 下验证
- 保持实现上的 `instance_id`、实例级配置、实例级存储和实例级日志边界

## 2. 非目标

本阶段明确不做：

- 多实例调度、聚合看板和跨实例命令分发
- Web UI 主线与远程操控
- 新的策略类型
- 在线参数编辑平台
- 多账户、多交易所扩展

## 3. 当前现状

截至 `2026-03-22`，仓库已经完成：

- `K4` 执行闭环与命令语义做实
- `K5` 网格策略与风控
- `K6` replay / paper / testnet 验证
- `K7` Web UI 查询模型与能力边界预备
- `K8` TUI 中英文切换

当前缺口不再是“能不能联调”，而是：

- mainnet 运行模式缺少明确准入保护
- 启动对账与异常拒绝启动边界还不够硬
- `7x24` 值守所需的健康状态、告警与恢复流程还未固化
- 当前网格策略距离长期实盘运行仍缺少参数护栏与更贴近实盘的风控收敛

## 4. 总体策略

推荐采用单主线串行推进：

1. `K9` 主网运行安全底座
2. `K10` 单实例 `7x24` 值守硬化
3. `K11` 单实例实盘策略收敛

原因：

- 没有 `K9`，系统不具备进入 mainnet 自动运行的准入资格
- 没有 `K10`，系统不具备持续值守资格
- `K11` 才是在安全与值守边界具备后，让策略更适合长期运行

## 5. K9：主网运行安全底座

### 5.1 目标

让一个已配置实例在 mainnet 下能够安全启动、完成启动对账、异常自动暂停，并在重启后恢复出明确运行结论。

### 5.2 范围

- 明确 `paper / testnet / mainnet` 运行级别
- mainnet 必须显式开启，默认不能误入
- 对账与风控动作按实例维度记录
- 当前只验收一个启用实例

### 5.3 核心能力

- 运行级别与准入保护
- 启动前校验：API 权限、symbol 元数据、价格精度、最小下单量、账户模式、杠杆、时钟偏差
- 启动对账：真实持仓、真实挂单、最近命令、策略状态、风控状态收敛
- 自动停机与硬保护：总仓位、挂单数、单次下单上限、连续失败、连续重建失败
- 恢复与可追溯：最近启动来源、对账结论、暂停原因、恢复原因

### 5.4 关键约束

- 对账失败默认进入暂停或拒绝启动，不能继续自动下单
- 保护动作必须有稳定原因码
- 所有关键判断都必须进入事件与日志，不能只存在于本地文本输出

## 6. K10：单实例 7x24 值守硬化

### 6.1 目标

让已经具备 mainnet 准入资格的实例，在本地部署和有人值守场景下具备持续运行、自动降级和可恢复能力。

### 6.2 核心能力

- 固化单实例部署、数据目录、日志目录与启动方式
- 重复启动保护与优雅停机
- 统一健康状态：`healthy / degraded / must_pause`
- 告警与值守信息：当前是否自动运行、为什么暂停、最近异常、是否需要人工介入
- 常见故障恢复路径：断网、用户流失效、交易所短时失败、服务重启、数据库恢复

### 6.3 关键约束

- `must_pause` 命中时必须自动停止新增策略动作
- 恢复后必须回到“可继续运行”或“已暂停待确认”的明确状态
- 告警先面向本地值守，不扩到远程操控

## 7. K11：单实例实盘策略收敛

### 7.1 目标

把当前“已经能运行的网格策略”收敛成更适合单实例 mainnet 长时间运行的策略。

### 7.2 核心能力

- 参数护栏：危险参数拦截、允许范围、风险提示
- 实盘风控收敛：单日亏损、连续异常成交、连续重建失败、异常波动暂停新挂单
- 策略状态可解释：为什么 `active / occupied / pending_rebuild / paused`
- 复盘能力：命令、成交、风控和策略状态变化进入同一可追溯链路

### 7.3 关键约束

- 风控动作必须与执行层、恢复层保持一致
- 启动前应拦截明显危险配置，不能把问题留到运行时
- 退出标准以稳定性、可控性、可追溯性为先，不以收益优化为先

## 8. 与多实例线程的边界

当前仓库另有线程推进“同一环境可承载多个实例”。本主线与其关系如下：

需要对齐：

- `instance_id` 或等价实例键的配置入口
- 实例级数据目录、数据库命名和日志上下文
- 实例级健康状态、事件与告警字段

当前先忽略：

- 多实例聚合看板
- 多实例批量操作
- 多实例联动风控
- 多实例资源竞争协调

## 9. 验收策略

继续按测试优先推进：

- 先定义或补齐验收测试
- 再补单元测试、集成测试和本地 E2E
- 每个里程碑都要有清晰退出标准

本路线的共同验收口径是：

- 一个已配置实例可以安全进入 mainnet 自动运行
- 关键异常会自动暂停而不是继续盲跑
- 本地值守者可以快速判断当前状态、风险和恢复动作
- 重启后能恢复出明确结论并追溯关键原因
