# Track 参数调试工作台设计

## 1. 目标

新增一个独立于 `poise-tui` 的桌面参数调试工作台，用于加载用户目录中的外部 TOML 配置文件，选择单个 `track` 做交互式试算，并基于 Binance 最新价格实时展示关键风险与调仓指标。

第一版目标是让用户可以更直观地回答下面这些问题：

- 当前 `min_rebalance_units` 对应多少价格波动
- 当前价格到风险边缘的带内理论损失是多少
- 零仓目标点到风险边缘的带内理论损失是多少
- 当前参数下，价格带、曲线、仓位规模和止损预算是否协调

这版工具只做本地试算，不回写原配置文件，不直接下单。

## 2. 选型结论

第一版采用 `Tauri` 独立桌面应用。

原因：

- 用户明确希望工具“像独立 app 一样方便使用”；桌面应用比手动起本地服务再开浏览器更符合这个目标
- 文件选择、剪贴板、窗口状态、本地草稿持久化都属于桌面壳擅长的能力
- 当前仓库技术栈以 Rust 为主，`Tauri` 更贴合现有能力边界
- 端口占用、服务存活、浏览器本地访问失败之类的问题可以直接设计掉

不选 `poise-tui` 内嵌的原因：

- 表单、图示、文件选择、复制导出在 TUI 中都更别扭
- 第一版目标是把调参体验做好，而不是最小集成成本

不选“独立部署意义上的前后端分离”的原因：

- 这个工具没有多客户端共享同一后端的需求
- 如果引入常驻本地 HTTP 服务，会额外增加端口、生命周期和调试复杂度
- 这类复杂度和核心价值无关，应该直接移除

## 3. 非目标

第一版明确不做：

- 不回写外部 TOML 文件
- 不直接修改仓库内现有配置流程
- 不接入 Binance 之外的自动行情源
- 不支持多 `track` 并排调参
- 不新增当前配置格式之外的“虚拟参数系统”
- 不增加新的硬止损价格字段或止损百分比字段
- 不直接接入交易所私有接口

## 3.1 体验与视觉质量要求

这不是一个只要“能算”就够的内部工具。第一版就要把交互质量和视觉质量当成正式约束。

要求：

- 界面层级必须清楚，用户一眼就能分辨：
  - 文件级操作
  - `track` 级操作
  - 当前最重要的风险结论
  - 可编辑参数
- 主图和关键指标必须形成稳定的视觉中心，不能让页面退化成普通参数表
- 交互要尽量减少打断：
  - 常用动作就近放置
  - 删除后可直接撤销
  - 草稿自动保存
  - 导出动作清晰，不制造“是否已保存”的歧义
- 视觉表达不能停留在“默认后台工具风格”
  - 布局需要有明确的视觉节奏
  - 信息密度高，但不能拥挤
  - 色彩、边框、留白和主次关系要有明确取舍
  - 审美目标是“专业、克制、但不普通”
- 图表和指标卡不是装饰，而是帮助用户做参数判断的核心界面
- 第一版即使不追求完整品牌化，也要避免做成“能用但很土”的工具

## 4. 用户流程

### 4.1 基本流程

1. 打开桌面参数调试应用
2. 选择一个外部 TOML 配置文件
3. 页面解析文件并列出其中全部 `[[tracks]]`
4. 用户选择一个 `track`
5. 页面显示该 `track` 的参数编辑器、主图和关键指标
6. 用户修改参数，页面立即重算
7. 草稿自动保存到本地会话
8. 用户按需复制：
   - 当前 `track`
   - 全部 `tracks`

### 4.2 草稿与会话

- 草稿只存在于本地调试会话中，不回写原文件
- 每个配置文件路径维护一份独立草稿
- 切换 `track` 不丢改动
- 刷新页面不丢改动
- 重新打开同一个配置文件时，默认恢复最近一次未导出的草稿

## 5. 页面结构

应用主窗口分成左右两栏。

### 5.1 左侧：文件与列表

左侧负责文件级操作和 `track` 列表管理。

包含：

- `选择配置文件`
- `撤销`
- `重做`
- `复制当前 Track`
- `复制全部 Tracks`
- 当前文件路径
- 当前是否存在未导出的草稿状态
- `空白新建 Track`
- `复制当前 Track`
- `track` 列表

每个列表项至少显示：

- `track_id`
- `symbol`
- 是否存在草稿修改

### 5.2 右侧：单 Track 调试工作台

右侧只服务一个当前选中的 `track`。

结构顺序：

1. 顶部关键指标卡
2. 主图：价格 / 仓位 / 风险
3. 参数编辑区

参数编辑区分成 4 组：

- 价格带
- 仓位与调仓
- 止损与带外策略
- 曲线与预览

## 6. Track 管理操作

### 6.1 新增

新增 `track` 支持两种方式：

- 空白新建
- 复制当前 `track`

### 6.2 删除

删除直接作用于当前草稿，不弹阻断式确认框。

删除后页面顶部显示短提示，例如：

- `已删除 silver，可撤销`

因为第一版具备完整撤销链，直接删再撤销的交互更顺。

## 7. 撤销 / 重做

第一版支持完整撤销 / 重做，但不记录每一个击键。

### 7.1 纳入历史的动作

- 新增 `track`
- 删除 `track`
- 复制 `track`
- 修改参数值
- 恢复当前 `track` 原始配置
- 恢复整份文件到加载时状态

### 7.2 不纳入历史的动作

- 输入框逐字输入过程
- 滑杆拖动过程中的每一帧
- 只读指标刷新

### 7.3 历史入栈时机

- 文本输入框：失焦或按 `Enter`
- 数字步进器：每点一次
- 滑杆：松手时
- 删除 / 新增 / 复制 / 恢复：动作完成时

### 7.4 历史范围

- 撤销历史只作用于当前已加载配置文件的本地会话
- 历史上限先固定为 `100` 步

## 8. 导出与复制

### 8.1 导出文本范围

导出文本不包含顶层 `exchange`、账户或其他全局配置，只包含 `[[tracks]]` 段。

### 8.2 两种复制动作

#### 复制当前 Track

只复制当前选中的一个 `[[tracks]]`。

#### 复制全部 Tracks

复制当前草稿中的全部 `[[tracks]]`。

导出总是基于“当前草稿状态”，而不是原始文件内容。

## 9. 参数编辑范围

第一版至少覆盖这些参数：

### 9.1 标识

- `track_id`
- `symbol`

### 9.2 价格带

- `lower_price`
- `upper_price`

### 9.3 仓位与调仓

- `long_exposure_units`
- `short_exposure_units`
- `notional_per_unit`
- `max_notional`
- `min_rebalance_units`
- `leverage`

### 9.4 止损与带外策略

- `out_of_band_policy`
- `daily_loss_limit`
- `total_loss_limit`

### 9.5 曲线

- `shape_family`

第一版不新增新的可导出止损字段。

### 9.6 当前不编辑字段的导出策略

第一版不追求 TOML 原样 round-trip。

规则：

- 导出只保证当前页面支持的字段
- 当前页面未暴露的字段，例如 `tick_timeout_secs`，第一版可以不导出
- 当前页面已暴露的字段，即使值等于系统默认值，第一版也应显式导出

## 10. 核心指标口径

所有页面指标都从同一份当前草稿状态计算，避免各区块口径不一致。

### 10.1 当前价格

- 默认来自 Binance 合约公共行情
- 按 `symbol` 直接拉取
- 默认使用最新成交价
- 自动刷新

若价格不可用：

- 页面明确显示不可用原因
- 仍允许继续调参和导出
- 提供一个“临时价格输入框”覆盖当前试算价格
- 临时价格只影响页面试算，不写入导出文本

### 10.2 最小步长对应价格

定义为：以当前价格为锚，目标仓位再变化 `min_rebalance_units` 所需的单边价格变化。

页面显示两项：

- `向上`
- `向下`

口径要求：

- `linear` 可以直接解析计算
- `inertial / responsive` 不使用线性近似，按当前价格做数值搜索
- 靠近边界时允许上下值不对称

### 10.3 当前价到风险边缘

定义为：从当前价格出发，沿对当前目标仓位更不利的方向移动到带内边缘，按策略目标曲线累计出的带内理论损失。

展示拆分：

- 带内理论损失
- 边缘平仓手续费估算
- 合计估算

若当前目标仓位接近 `0`，则同时显示上下两个方向。

### 10.4 零仓目标点到风险边缘

文案统一使用“零仓目标点”，不用“零轴”。

定义为：从 `desired_exposure = 0` 对应的价格点出发，沿更不利方向移动到带内边缘，按策略目标曲线累计出的带内理论损失。

作用：

- 不依赖当前市场位置
- 用于描述这组参数本身的激进程度

### 10.5 当前目标仓位

同时显示：

- `units`
- 名义价值
- 换算数量

### 10.6 其他辅助指标

第一版还显示这些只读指标：

- 价格带宽
- 带中点
- `1 unit` 对应价格变化
- `1 unit` 对应基础数量
- 最大目标名义仓位
- 当前目标名义仓位

## 11. 主图要求

主图是页面一等区域，不是附属小图。

主图必须基于真实试算曲线生成，不能使用仅供视觉展示的近似形状。

规则：

- 曲线采样必须直接来自当前页面统一试算模块
- 曲线口径必须与页面上的仓位、最小步长、风险指标使用同一套公式
- `linear / inertial / responsive` 的形状差异必须由真实公式产生，不能手工画模板曲线
- 不允许为了视觉顺滑另行改写曲线语义；如果需要平滑，只能在不改变真实取样点含义的前提下做显示层处理

主图至少表达：

- 当前价格带
- 当前曲线形状
- 当前价格位置
- 零仓目标点
- 当前价到风险边缘的方向

主图会随下面这些参数变更即时刷新：

- `lower_price`
- `upper_price`
- `long_exposure_units`
- `short_exposure_units`
- `shape_family`
- `min_rebalance_units`
- 当前试算价格

## 12. Tauri Rust 命令层职责

`Tauri` 的 Rust 侧只做环境适配，不承担复杂试算逻辑。

### 12.1 Rust 侧负责

- 读取外部 TOML 配置文件
- 解析配置并返回全部 `tracks`
- 拉 Binance 合约公共价格
- 返回原始配置文本
- 生成复制文本：
  - 当前 `track`
  - 全部 `tracks`
- 处理文件选择器、剪贴板和本地持久化这类桌面能力

### 12.2 Rust 侧不负责

- 不维护复杂的调参会话历史
- 不承担主要指标计算
- 不承担主图计算

## 13. 前端职责

前端是参数工作台本体，负责：

- 当前草稿状态
- 选中的 `track`
- 撤销 / 重做
- 自动保存
- 全部试算计算
- 主图和指标卡渲染
- 参数合法性提示

因此前端需要一个统一的纯计算模块，所有指标都从该模块导出，避免重复实现。

## 13.1 内部分层原则

虽然最终打包形态是桌面应用，但代码结构仍然按前端 / Rust 命令层分层：

- 前端拥有交互和试算知识
- Rust 命令层拥有文件、行情、剪贴板和系统能力知识

这是一种“单应用、内部分层”的设计，不是部署意义上的前后端分离。

## 14. 错误与边界处理

### 14.1 文件加载错误

- TOML 无法解析时，明确显示错误位置和原因
- 文件结构缺字段时，标记对应 `track` 不可试算

### 14.2 Binance 价格错误

- symbol 不存在：显示 `Binance 不支持该 symbol`
- 网络失败：显示 `当前价格不可用`
- 价格不可用不阻断其他调参功能

### 14.3 参数错误

对无效输入即时提示，不阻断继续编辑：

- `lower_price >= upper_price`
- 多空容量同时为 `0`
- 非法的损失预算
- 其他违反当前项目约束的输入

## 15. 设计取舍

### 15.1 为什么不直接回写文件

- 用户明确要求第一版不写回原文件
- 先把试算和比较体验做顺，再决定是否允许安全写回

### 15.2 为什么导出只保留 `[[tracks]]`

- 用户主要想复用和迁移的是轨道参数
- 顶层 `exchange` 等全局配置不属于这次调参的核心产物

### 15.3 为什么不做多 Track 并排

- 当前核心任务是把单 `track` 的调参体验做顺
- 并排比较会显著增加布局、状态和导出语义复杂度

### 15.4 为什么选 Tauri 而不是常驻本地服务

- 用户体验更接近真正可用的独立工具
- 文件与剪贴板能力更自然
- 去掉端口和服务生命周期之后，错误面更小
- 当前问题的复杂度应该压到应用内部，而不是暴露成运行环境要求

## 16. 验收标准

第一版完成后，用户应能做到：

1. 选择任意一个外部配置文件并加载
2. 在页面中选择某个 `track`
3. 修改核心参数并看到主图和指标即时变化
4. 查看：
   - 最小步长对应价格
   - 当前价到风险边缘
   - 零仓目标点到风险边缘
5. 新增、复制、删除 `track`
6. 对上述动作执行撤销 / 重做
7. 刷新页面后继续上次草稿
8. 复制当前 `track`
9. 复制全部 `tracks`
10. 在 Binance 价格不可用时仍继续试算
