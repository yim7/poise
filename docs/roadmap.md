# 网格平台 Roadmap

本文档只定义项目的阶段目标、依赖关系和退出标准。
具体任务拆分、测试矩阵和并行方式统一放在 [`plan.md`](plan.md)。

## 2. 横切轨道

下面三条轨道贯穿所有阶段，不单独作为线性阶段推进：

- 客户端轨道：TUI 跟随服务端能力演进，未来 Web UI 复用同一控制面。
- 协议轨道：Rust 协议模型、HTTP / WS envelope、兼容策略和重连语义持续维护。
- 测试与可观测性轨道：内核测试、协议测试、本地 E2E、回放验证和诊断能力持续补齐。

## 3. 阶段 0：Rust 基线与协议基础

### 目标

建立双 crate 工程形态，并打通当前控制面与客户端联调。

### 阶段结果

- [`service/`](../service/) 与 [`tui/`](../tui/) 形成稳定基线
- 协议模型与语义文档建立
- `snapshot + ws + command ack` 最小闭环打通
- 本地 E2E 有基础覆盖

### 退出标准

- `service` 和 `tui` 可分别运行
- 协议与本地 E2E 有测试基线
- TUI 可在真实服务骨架上运行

## 4. 阶段 1：服务端内核重构

### 目标

把当前内存态控制面重构为可长期承载交易逻辑的服务端内核。

### 阶段结果

- `service` 内部分出 `control_plane / application / kernel`
- 状态变更统一进入单写者路径
- 命令入口、领域事件出口和查询模型边界清晰
- 对外 HTTP / WS 协议保持兼容

### 退出标准

- 服务端不再依赖多处直接改共享状态
- 命令、行情、系统事件都通过内核统一收敛
- 查询路径和变更路径清晰分离

## 5. 阶段 2：持久化与恢复

### 目标

让 `service` 从内存骨架进入可恢复、可审计状态。

### 阶段结果

- SQLite 存储层建立
- 当前状态表与审计事件表成型
- 启动恢复流程稳定
- 客户端在恢复与重连下仍能维持一致性

### 退出标准

- `service` 重启后可恢复最近运行态
- 命令链路有完整审计
- TUI 不依赖连续不断流

## 6. 阶段 3：市场与账户接入

### 目标

接入 Binance 的市场与账户基础数据，为后续执行与策略打底。

### 阶段结果

- 元数据同步进入 `service`
- 市场数据流和用户流可接入
- 连接健康状态进入内核模型
- TUI 能看到真实连接和 session 状态

### 退出标准

- `service` 能稳定获取真实市场状态
- `session_state` 不再是假数据
- 连接健康状态可被 TUI 明确感知

## 7. 阶段 4：执行闭环与命令语义做实

### 目标

把命令从控制面回包推进到真实执行闭环。

### 阶段结果

- 执行适配层建立
- 命令与下游 order/fill 形成关联
- `ack / failure / timeout` 有统一语义
- open orders 与 fills 进入统一命令轨迹

### 退出标准

- 命令不再只是内存状态切换
- ack / failure / timeout 有明确业务意义
- open orders 与 fills 可与命令轨迹关联

## 8. 阶段 5：网格策略与风控

### 目标

让 `service` 真正承载网格运行逻辑。

### 阶段结果

- 网格配置和状态机成型
- 风控阈值、breaker 和风险事件进入统一模型
- TUI 的 `Grid` 页面展示策略状态而非订单镜像

### 退出标准

- `service` 内部有真实策略状态机
- `Grid` 页面显示的是策略模型而非订单镜像
- 风险事件可观测、可操作

## 9. 阶段 6：回放 / paper / testnet 验证

### 目标

建立可重复验证链路，把系统从能运行推进到能被证明。

### 阶段结果

- replay / paper 验证路径建立
- fake service / fake transport 集成验证成型
- testnet 冒烟流程可重复运行
- 运维与验证手册形成基线

### 退出标准

- 本地可重复回放
- testnet 可跑通基本操作
- 关键路径有端到端验证

## 10. 阶段 7：Web UI 就绪与多实例预备

### 目标

让控制面具备承载更多客户端形态的能力。

### 阶段结果

- 面向前端的查询模型整理完成
- 分页、过滤和 `instance_id` 预留清晰
- 局域网部署与简单认证边界明确

### 退出标准

- TUI 与 Web UI 能共用一套控制面
- 控制面无需为新客户端重写

## 11. 阶段 8：TUI 中英文切换

### 目标

让 `tui` 的本地界面文案支持 `zh-CN / en-US` 双语切换，同时保持服务端接口、协议字段和服务端返回文本不变。

### 阶段结果

- `tui` 新增统一本地化文案入口，不再在多个模块散落写死显示文本
- 启动配置支持指定默认语言
- 运行时支持热切换语言
- 英文默认行为保持稳定
- 中文窄屏布局有快照与回归保护

### 退出标准

- `GRID_PLATFORM_UI_LOCALE` 可指定启动语言
- 运行时按键可在中英文之间切换
- 服务端返回文本继续原样显示
- 关键页面在中文模式下不出现明显布局错位或关键信息丢失
- `cargo test -p grid-platform-tui` 与 `cargo test` 通过

## 12. 阶段 9：主网运行安全底座

### 目标

让一个已配置实例具备进入 mainnet 自动运行的最低安全资格。

本阶段采用“实例感知，单实例验收”的边界：

- 平台实现继续保留实例级配置、存储、日志与事件上下文
- 当前主线只验收一个启用实例
- 不把多实例运营能力放进本阶段退出标准

### 阶段结果

- `paper / testnet / mainnet` 运行级别明确分离
- mainnet 启动前校验形成统一准入门槛
- 启动对账可对真实持仓、真实挂单、策略状态与风控状态给出明确结论
- 异常时可自动暂停策略并阻止继续新增挂单
- 重启后可恢复最近关键运行结论并追溯原因

### 退出标准

- mainnet 必须显式开启，默认不会误入
- 启动前关键校验失败时，实例拒绝进入自动运行
- 对账失败时，实例进入暂停或拒绝启动，而不是继续自动下单
- 硬保护命中时，系统自动暂停策略并保留明确原因码
- 配置、日志、持久化与事件上下文均保留实例边界

## 13. 阶段 10：单实例 `7x24` 值守硬化

### 目标

让已经具备 mainnet 准入资格的实例，具备本地长期值守所需的运行、告警与恢复能力。

### 阶段结果

- 单实例部署、数据目录、日志目录与启动方式固化
- 重复启动保护与优雅停机路径明确
- 统一健康状态和自动降级语义形成稳定模型
- 本地值守所需的关键信息与告警入口补齐
- 常见故障有稳定恢复路径与恢复后状态结论

### 退出标准

- 单实例可持续运行并具备明确的重复启动保护
- 关键故障会触发 `degraded` 或 `must_pause`，而不是继续盲跑
- 恢复后系统回到“可继续运行”或“已暂停待确认”的明确状态
- 值守者能快速判断是否在自动运行、为什么暂停、是否需要人工介入

## 14. 阶段 11：单实例实盘策略收敛

### 目标

把当前网格策略从“已经能运行”推进到“更适合单实例 mainnet 长时间运行”。

### 阶段结果

- 参数护栏与危险参数拦截成型
- 更贴近实盘的风险规则与动作语义稳定
- 策略状态与最近关键变化可解释、可查询、可展示
- 命令、成交、风险与策略状态变化形成统一复盘链路

### 退出标准

- 危险参数可在启动前被拦截或明确标记高风险
- 实盘常见异常会触发明确风险动作
- 当前策略状态与最近关键变化可追溯
- 小范围 mainnet 连续运行后，出现问题时能停、能查、能恢复

## 15. 当前优先级

当前主线已经明确为 `K10`：单实例 `7x24` 值守硬化。

原因：

- 截至 `2026-03-22`，`K9` 已完成并通过验收，当前瓶颈已从“主网自动运行资格”切到“本地长期值守能力”
- 用户当前目标是单实例 mainnet 自动运行与本地值守，不需要远程操控
- `K10` 已完成里程碑设计，下一步应先补值守健康状态、重复启动保护与告警测试，再进入实现
- 仓库已有多实例相关并行线程，但当前主线可以按“实例感知、单实例验收”推进，不必等待多实例运营能力成熟
