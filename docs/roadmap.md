# 网格平台 Roadmap

本文档只定义项目的阶段目标、依赖关系和退出标准。
具体任务拆分、测试矩阵和并行方式统一放在 [`plan.md`](plan.md)。

## 1. 路线概览

| 阶段 | 核心能力 | 关键依赖 | 当前状态 |
|---|---|---|---|
| 阶段 0 | Rust 基线与协议基础 | 无 | 已完成 |
| 阶段 1 | 服务端内核重构 | 阶段 0 | 已完成 |
| 阶段 2 | 持久化与恢复 | 阶段 1 | 已完成 |
| 阶段 3 | 市场与账户接入 | 阶段 2 | 已完成 |
| 阶段 4 | 执行闭环与命令语义做实 | 阶段 3 | 已完成 |
| 阶段 5 | 网格策略与风控 | 阶段 4 | 已完成 |
| 阶段 6 | 回放 / paper / testnet 验证 | 阶段 5 | 未开始 |
| 阶段 7 | Web UI 就绪与多实例预备 | 阶段 4 到 6 稳定 | 未开始 |

## 2. 横切轨道

下面三条轨道贯穿所有阶段，不单独作为线性阶段推进：

- 客户端轨道：TUI 跟随服务端能力演进，未来 Web UI 复用同一控制面。
- 协议轨道：Rust 协议模型、HTTP / WS envelope、兼容策略和重连语义持续维护。
- 测试与可观测性轨道：内核测试、协议测试、本地 E2E、回放验证和诊断能力持续补齐。

## 3. 阶段 0：Rust 基线与协议基础

### 目标

建立双 crate 工程形态，并打通当前控制面与客户端联调。

### 阶段结果

- [`service/`](../service/) 与 [`tui/`](../tui/) 形成稳定基线
- 协议模型与语义文档建立
- `snapshot + ws + command ack` 最小闭环打通
- 本地 E2E 有基础覆盖

### 退出标准

- `service` 和 `tui` 可分别运行
- 协议与本地 E2E 有测试基线
- TUI 可在真实服务骨架上运行

## 4. 阶段 1：服务端内核重构

### 目标

把当前内存态控制面重构为可长期承载交易逻辑的服务端内核。

### 阶段结果

- `service` 内部分出 `control_plane / application / kernel`
- 状态变更统一进入单写者路径
- 命令入口、领域事件出口和查询模型边界清晰
- 对外 HTTP / WS 协议保持兼容

### 退出标准

- 服务端不再依赖多处直接改共享状态
- 命令、行情、系统事件都通过内核统一收敛
- 查询路径和变更路径清晰分离

## 5. 阶段 2：持久化与恢复

### 目标

让 `service` 从内存骨架进入可恢复、可审计状态。

### 阶段结果

- SQLite 存储层建立
- 当前状态表与审计事件表成型
- 启动恢复流程稳定
- 客户端在恢复与重连下仍能维持一致性

### 退出标准

- `service` 重启后可恢复最近运行态
- 命令链路有完整审计
- TUI 不依赖连续不断流

## 6. 阶段 3：市场与账户接入

### 目标

接入 Binance 的市场与账户基础数据，为后续执行与策略打底。

### 阶段结果

- 元数据同步进入 `service`
- 市场数据流和用户流可接入
- 连接健康状态进入内核模型
- TUI 能看到真实连接和 session 状态

### 退出标准

- `service` 能稳定获取真实市场状态
- `session_state` 不再是假数据
- 连接健康状态可被 TUI 明确感知

## 7. 阶段 4：执行闭环与命令语义做实

### 目标

把命令从控制面回包推进到真实执行闭环。

### 阶段结果

- 执行适配层建立
- 命令与下游 order/fill 形成关联
- `ack / failure / timeout` 有统一语义
- open orders 与 fills 进入统一命令轨迹

### 退出标准

- 命令不再只是内存状态切换
- ack / failure / timeout 有明确业务意义
- open orders 与 fills 可与命令轨迹关联

## 8. 阶段 5：网格策略与风控

### 目标

让 `service` 真正承载网格运行逻辑。

### 阶段结果

- 网格配置和状态机成型
- 风控阈值、breaker 和风险事件进入统一模型
- TUI 的 `Grid` 页面展示策略状态而非订单镜像

### 退出标准

- `service` 内部有真实策略状态机
- `Grid` 页面显示的是策略模型而非订单镜像
- 风险事件可观测、可操作

## 9. 阶段 6：回放 / paper / testnet 验证

### 目标

建立可重复验证链路，把系统从能运行推进到能被证明。

### 阶段结果

- replay / paper 验证路径建立
- fake service / fake transport 集成验证成型
- testnet 冒烟流程可重复运行
- 运维与验证手册形成基线

### 退出标准

- 本地可重复回放
- testnet 可跑通基本操作
- 关键路径有端到端验证

## 10. 阶段 7：Web UI 就绪与多实例预备

### 目标

让控制面具备承载更多客户端形态的能力。

### 阶段结果

- 面向前端的查询模型整理完成
- 分页、过滤和 `instance_id` 预留清晰
- 局域网部署与简单认证边界明确

### 退出标准

- TUI 与 Web UI 能共用一套控制面
- 控制面无需为新客户端重写

## 11. 当前优先级

当前最值得投入的顺序是：

1. 阶段 6 回放 / paper / testnet 验证
2. 阶段 7 Web UI 就绪与多实例预备

原因：

- 执行闭环、策略状态机和风险事件已经具备稳定基线，下一步需要补可重复验证证据
- Web UI 需要建立在稳定的查询模型和验证链路之上，不宜先行
